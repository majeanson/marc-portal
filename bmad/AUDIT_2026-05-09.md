# Marc Portal — Where We Are, What's Missing, What's Next

**Date:** 2026-05-09
**Method:** five parallel reviews — product/feature, code quality, security, DX/ops, UX/content. Each was given the full repo and ~5–8 minutes to walk it. Findings below are the synthesis, not the agents' raw output.

---

## TL;DR — five things to know at a glance

1. **The bedrock rule is wallpaper, not code.** The "1 active build + 1 in triage, no exceptions" rule is the single load-bearing constraint of the entire practice. It is not enforced anywhere. The hero CTA flips a UI label; the intake form keeps submitting; `POST /api/sessions` creates the row regardless. The system that's supposed to protect family-time has zero structural backing.

2. **The cornerstone CTA is a stub.** The Sunday Night Dread demo references three audio files (`/audio/snd-tuesday-tremblay.mp3` etc.) — `public/audio/` does not exist. The parser is hard-coded to FR transcripts even when `lang === 'en'`. The buyer-facing tenanted version of SND (`SndApp.tsx`) doesn't even call the parser. The showcase that's supposed to *be the proof* is missing the proof.

3. **Two security bugs are launch-blockers, not someday-fixes.** Magic-link tokens are stored plaintext in D1 (DB snapshot leak = free account takeover for any pending token). `SESSION_SECRET` has no boot guard — if the env var is missing in any environment, HMAC silently encodes the literal string `"undefined"` and every cookie can be forged. Both are small, free fixes. Neither has been done.

4. **Vision is resolved: solo practice, small client count, Marc operates everything through the portal.** The last 30 commits drifted toward a multi-tenant marketplace shape (fleet foundation, operator console, buyer admin, tenant-scoped handlers, productized templates). That is now confirmed *not* the product. The fleet/tenant code stays as architecture (one tenant — Marc) but is not a marketplace. Buyer-admin / operator-console / Stripe / multi-customer support work is **not** on the roadmap. Downtime tolerance is high (most projects can sit unattended for days; "huge downtimes aren't an issue"). This unlocks decisions: skip paid observability, keep ops minimal, cut the 90-day SaaS branch entirely.

5. **The codebase is healthier than its size suggests.** Strict TS, zero `any` outside one JSON-import assert, real auth/tenant providers, end-to-end typed API client, parameterized SQL throughout, soft-delete + status history, lazy routes. Real debt is small and surgical (4 duplicated `loadSession` calls, hand-mirrored client/server types, FR/EN route tree literally doubled). The boring-tech mandate is honored — 3 runtime deps. *This is a good foundation; spend hours on the right things.*

## Vision (resolved)

Decided 2026-05-09: **solo practice, small client count, Marc operates everything through the portal himself.** The portal exists to minimize Marc's time, not to scale to N tenants. Most projects have minimal impact and tolerate "huge downtimes" — uptime SLA is informal, paid observability is overkill, multi-customer support infrastructure is not needed.

Implications that follow from this decision:

- **Fleet/tenant primitives stay as architecture, not as a marketplace.** The code Marc already wrote (`feat-fleet-foundation`, `feat-platform-spine`, `feat-runtime-theme-editor`, `feat-custom-domain-onboarding`) is kept — it costs nothing to leave in place and it might be useful for Marc's own per-engagement white-labeling. But buyer-admin / operator-console / Stripe / billing UI are **not** roadmap items. The "Stripe arrive bientôt" stubs (`AdminBilling.tsx:33`, `AdminTeam.tsx:20`) stay as stubs or get hidden from nav.
- **Downtime tolerance is high.** No paid Sentry, no UptimeRobot Pro, no custom alerting. A free `/api/health` ping + a once-a-day check is plenty. The "11pm runbook" matters less for this practice than for a SaaS — if prod is down at 11pm, it's down until tomorrow morning.
- **No multi-tenant isolation hardening pass.** Tenant resolution code stays correct (it already is) but doesn't get an exhaustive cross-tenant-leak test suite — there is one tenant operationally.
- **Support load is single-channel.** All support comes back to Marc, through the portal's own message threads. No ticketing system, no email forwarding, no Slack.
- **The bedrock 1-active rule still rules.** Even with "small client count," the cap is the lever that protects family-time. It's still the single most important thing to enforce in code.

The original `feature.json` already describes this product accurately. It does not need to be rewritten. The fleet code is keepable as scaffolding; the only docs change needed is a one-line note to future-Marc-or-Claude in `feature.json`'s `analysis`: *"Fleet/tenant primitives exist as architecture for Marc's own per-engagement isolation; this is not a marketplace and there are no paying buyers."*

## Five things to fix before any real client touches this

### 1. Hash magic-link tokens at rest (~30 min, free, security-critical)

`functions/api/auth/request-link.ts:58-66` writes the raw 32-byte token to D1; `verify.ts:33-37` queries by that exact value. Any DB read (a backup, an exported snapshot, a misconfigured admin query) becomes a free account takeover for every unconsumed token.

Fix: generate the random bytes, hash with SHA-256, store the hash. Send only the plaintext in the URL. Hash on lookup.

### 2. Add a `SESSION_SECRET` boot guard (~15 min, free, security-critical)

`functions/_lib/auth.ts:19` does `TextEncoder().encode(env.SESSION_SECRET)`. If `SESSION_SECRET` is missing or empty, `encode(undefined)` produces the bytes for the literal string `"undefined"` — a publicly-known HMAC key. Any environment misconfiguration silently downgrades cookie integrity to "anyone can forge sessions". A type signature of `string` does not catch this; runtime does not catch this; CI does not catch this.

Fix: `if (!env.SESSION_SECRET || env.SESSION_SECRET.length < 32) throw new Error('SESSION_SECRET missing or too short')` at the top of every handler that touches auth. Or in the middleware. Fail loudly.

### 3. Enforce the 1-active cap server-side (~4–6h)

`POST /api/sessions` (`functions/api/sessions/index.ts:78`) creates a session with no count check. `public/data/capacity.json` is a hand-edited static fixture. The hero label flips, the intake notice appears, but the form continues. **The single load-bearing rule of the system has zero structural backing.**

Fix: add `countActiveAndTriage(env, tenantId)` to `functions/_lib/sessions.ts`; gate `POST /api/sessions` and the `PATCH status → active` transition on it; return `409 + waitlist routing` when at cap. Delete `public/data/capacity.json` and have the `<CapacityCounter>` read from `/api/capacity` (which already exists) so there's only one source of truth.

This is the rule that, if broken, breaks the bedrock value (family-first). If you ship anything else from this audit, ship this.

### 4. Wire D1 migrations into the deploy pipeline (~1h)

`.github/workflows/deploy.yml` deploys code but never runs `wrangler d1 migrations apply marc-portal-db --remote`. Today's flow is: code lands, deploy succeeds, server 500s until Marc remembers to run migrations manually. Migration `0007_attachments` may already be at risk of this exact bug.

Fix: add a step before the wrangler-action call: `npx wrangler d1 migrations apply marc-portal-db --remote --yes` (idempotent — no-op if already applied). The first time you forget on a real client → prod outage at 11pm.

### 5. Set D1 region / write a privacy page (Loi 25 — ~half day)

The footer says "Hébergé au Canada · Loi 25 · OQLF" but:

- `wrangler.toml` has no `location_hint` on the D1 binding — CF auto-routes, likely to US East
- There is no `/confidentialite` (or `/en/privacy`) route in `App.tsx`
- No DPO contact, no breach process, no portability endpoint, no self-serve "delete my data"
- Visitor email addresses appear in notification email subject lines (`functions/_lib/email.ts:71,162,189`), leaking client identity via lock-screen previews

Loi 25 explicitly requires a published privacy policy with a designated person. The footer claim is currently hollow. A Quebec notary or accountant — exactly the kind of client this practice targets — will check.

Fix: add `location_hint = "enam"` to the D1 binding; write a single Markdown page at `/confidentialite` covering the eight Loi 25 obligations (residency, consent, access, rectification, erasure, portability, retention, breach contact); strip identifying info from email subjects, keep it in the body.

## Things to NOT do (explicit don'ts so hours don't get burned)

- **Don't introduce a Page-shell HOC.** `MePortal`, `SessionPage`, `Login`, etc. all wrap `<><Header/><main>...</main><Footer/></>` with subtle differences. Consolidating forces prop bloat. Keep duplicated.
- **Don't introduce a `t('key')` i18n library.** The current `COPY = { fr, en } as const` idiom is type-safe (TS infers keys), zero-runtime, and never breaks. A global string registry is exactly what the boring-tech mandate rejects.
- **Don't introduce Sentry / Rollbar / paid observability yet.** Free CF Workers Logs + a health endpoint cover everything at this scale.
- **Don't refactor the warm-amber → sage-cream palette twice.** The `og-image.svg` is the only place still on the old palette. One file to swap, then leave it alone.
- **Don't add Stripe.** Vision is solo practice — invoicing happens out-of-band (e-Transfer, manual invoice email). The "Stripe arrive bientôt" stub gets hidden from nav, not implemented.
- **Don't harden multi-tenant isolation.** One tenant operationally. Cross-tenant leak tests aren't worth the hours when there's nothing to leak between.
- **Don't write the buyer-admin / operator-console UI.** Those surfaces are stubs and stay stubs. If Marc ever opens `/admin/equipe` accidentally, that's fine — nobody else will.
- **Don't add paid observability.** Free CF Workers Logs + a single `/api/health` endpoint is the entire observability budget. Downtime is acceptable.
- **Don't consolidate `MePortal` SessionRow + `AdminInbox` row.** Different audiences, different motion.
- **Don't add a 4th admin route under the current doubled FR/EN tree.** Refactor item 13 above first.

## The 11pm runbook (5 most likely production failures)

If prod breaks at 11pm and your brain is at 30%, these are the suspects in order:

1. **Magic-link emails not arriving.**
   - First check: Resend dashboard for free-tier ceiling (100/day) or sender-domain block.
   - Second check: `functions/_lib/email.ts:6` uses `onboarding@resend.dev` (Resend's shared domain — no SPF/DKIM, may be greylisted).
   - Quick mitigation: tell user to check spam.
   - Proper fix: verify a custom sender domain on Resend.

2. **`/api/capacity` or `/api/sessions` returns 500.**
   - First check: a migration didn't run on prod. `npx wrangler d1 migrations apply marc-portal-db --remote`.
   - The audit log endpoint and capacity endpoint are the two that fail loudest if the schema is behind.

3. **Every request 404s.**
   - `functions/_middleware.ts:60-67`: a domain not in `tenant_domains` table 404s every request.
   - Fix: `wrangler d1 execute marc-portal-db --remote --command "INSERT INTO tenant_domains..."` per `functions/db/migrations/README.md`.

4. **Build fails in CI with `npm ci` lockfile error.**
   - This is the `@emnapi/*` Windows-vs-Linux optionalDeps thing that bit us today.
   - Fix: regenerate the lockfile with `npm install --include=optional --package-lock-only` and re-push.

5. **Deploy succeeded but pages serve old code.**
   - CF Pages caches aggressively. `_headers` correctly sets `Cache-Control: public, max-age=31536000, immutable` for `/assets/*` (content-hashed — fine).
   - Check that `index.html` is not cached. If stale, purge cache from CF dashboard.
   - Roll back: CF Pages "Deployments" tab → "Rollback" on the prior good deployment. (D1 schema is forward-only; old code against new schema is usually fine.)

## Honest assessment, in one paragraph

This is a thoughtfully-built side practice in a healthy state, with a coherent design philosophy that's mostly held up under code-level scrutiny. The boring-tech mandate is real and is doing what it was supposed to do — three runtime deps, no UI framework, no state library, strict TS, end-to-end typed API client, and a small `_lib/` that behaves like an in-house framework without the framework tax. **The gap between what the homepage promises and what the cornerstone proof actually does is the largest single risk to the practice.** The SND demo references audio files that don't exist; the showcase wall has one entry; the 1-active cap that the homepage advertises as inviolable is a UI label, not a constraint. Two security bugs (plaintext magic-link tokens, no `SESSION_SECRET` boot guard) are launch-blockers but cheap to fix. With the vision now resolved — solo practice, small client count, downtime-tolerant — none of the items in this audit are existential. *All* are surgical fixes that fit inside the 5–15h/week budget. ## Session plan

Five evening sessions of ~3–4h each. Total ~18 hours = 2–3 weeks at 5–15h/week. Each session is independently shippable and leaves the portal in a strictly-better state — if Marc gets to Session 2 and stops, the codebase is still safer and more honest than today.

The order is **deliberately sequenced by reversibility**: security and the cap go first (cheap, hard to undo, protect the bedrock), then the credibility surface (visible to prospects), then compliance (required before signing), then operational nice-to-haves, then polish. Skip Session 6 unless you have leftover hours.

The "things that prevent catastrophic embarrassment" session. Nothing in this session is visible to a visitor; all of it is the floor under everything else.

### Session sequencing rationale

- **1 before 2** — security and the cap are floor work; everything visible is built on top of them. If Session 2's homepage refactor introduces a regression, you want the cap and HMAC fixes already shipped.
- **2 before 3** — Session 2 lifts the homepage conversion floor; Session 3 doesn't change conversion but is required before signing a paying client. If outreach happens between sessions 2 and 3 and someone wants to sign immediately, you can sprint 3 in a single evening.
- **3 before 4** — Session 4 (triage digest, intake drafts) only matters once you have inbound; Session 3 is what makes inbound legal.
- **4 before 5** — Session 5's tests cover the auth code that Session 1 hardened; doing them after means you've had 2 weeks of "is this actually working in prod" before encoding the behavior in tests, which catches bugs the tests would have asserted away.
- **6 last** — purely cosmetic; never blocks anything.
