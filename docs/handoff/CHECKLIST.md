# Handoff Checklist — Marc Portal

> Internal mirror of the public `/handoff/checklist` page. The page is the
> source of truth for clients; this file is for Marc's offline reference and
> for clients who want a printable / non-JS version.
>
> **Public URL:** <https://marc-portal.pages.dev/handoff/checklist> (FR) /
> <https://marc-portal.pages.dev/en/handoff/checklist> (EN)

This is the exact procedure executed at engagement close. The client can read
it ahead of time, ask questions, and verify each step happened. There is no
hidden phase — every transfer is documented here.

---

## Legend

- **Mode « Tout à toi »** — step executed at delivery day (D-day).
- **Mode « Je m'en occupe »** — step executed the day the client requests the
  transfer (on demand), or after 90 days of dormancy via the kill-switch.

---

## 1. Pre-confirmation (D-7)

- Marc sends a recap email: chosen mode, target accounts, transfer date.
- Client replies with: GitHub username, preferred registrar (OVH / Namecheap /
  other), Cloudflare email (existing or new).
- Marc creates a shared 1Password vault (or encrypted Bitwarden note) with
  every credential — read-only access until D-day.

## 2. GitHub repo

- `git tag v1.0-handoff` on the last deployed commit, push the tag.
- Transfer repo ownership: Settings → Transfer ownership → target account.
- Client accepts transfer in GitHub notifications (within 24h).
- Marc removes himself from collaborators (Settings → Collaborators).
- **Verification:** `git clone git@github.com:<client>/<repo>.git` succeeds.

## 3. Domain name

- **Case A** — domain already in client's name: no transfer. Marc removes
  admin access at the registrar.
- **Case B** — domain in Marc's name: Marc initiates the transfer at his
  registrar, hands the auth-code to the client, client accepts at their
  chosen registrar (5–7 days).
- Site stays online during transfer (DNS keeps pointing).
- After transfer: client updates WHOIS with their contact details.

## 4. Hosting (Cloudflare Pages)

- **Option A** — direct transfer: client creates a Cloudflare account (free),
  Marc moves the project to their account (Settings → Move).
- **Option B** — rebuild on client's side: Marc gives `wrangler.toml`,
  bindings (D1, R2), and the `npx wrangler pages deploy` command.
- All env vars (SESSION_SECRET, RESEND_API_KEY, etc.) are regenerated
  (see §7) and entered into the client's Cloudflare dashboard.
- **Verification:** `curl https://<client-domain>/api/health` returns
  `{ "ok": true }`.

## 5. Database (Cloudflare D1)

- Dump: `wrangler d1 export marc-portal-db --remote --output handoff.sql`.
- Encrypt with client's GPG public key (or symmetric password via 1Password).
- Client creates D1 in their account: `wrangler d1 create marc-portal-db`.
- Load dump:
  `wrangler d1 execute marc-portal-db --remote --file=handoff.sql`.
- Update `database_id` in `wrangler.toml`.
- **Verification:**
  `wrangler d1 execute marc-portal-db --remote --command "SELECT COUNT(*) FROM users"`
  returns the expected row count.

## 6. Email service (Resend)

- Transfer Resend account ownership (Settings → Team → Transfer).
- Client verifies their domain in Resend; SPF, DKIM, DMARC published in DNS.
- Marc revokes old API keys.
- Client generates a new `RESEND_API_KEY` and adds it to Cloudflare env vars.
- **Verification:** sign in to the portal, request a magic link — email
  arrives.

## 7. Secret rotation

All secrets regenerated at handoff, regardless of mode:

- `SESSION_SECRET` — `openssl rand -hex 32` (≥ 32 chars).
- `RESEND_API_KEY` — newly generated by client in Resend.
- `DIGEST_TOKEN` (if used) — newly generated, shared via 1Password.

Marc then:

- Deletes local copies of every secret.
- Revokes all admin access on Cloudflare, GitHub, Resend, registrar.
- The audit log (`/admin/audit`) shows the last entry signed by Marc at
  delivery — a permanent reference point.

## 8. Dormancy procedure (kill-switch)

If Marc becomes unreachable for **90 consecutive days** — no reply to 3
emails spaced at least 30 days apart, sent to
<marc.jeanson92@gmail.com> — the client can execute this procedure
unilaterally to take full ownership:

1. **Acknowledgment.** Send a final email titled
   `DORMANCY 90 days — unilateral ownership claim in progress` to
   <marc.jeanson92@gmail.com>, CC a witness (lawyer, accountant, technical
   friend).
2. **GitHub reset.** Open a GitHub Support ticket with proof of the original
   contract + log of unanswered emails. GitHub transfers ownership within
   ~30 days.
3. **Cloudflare reset.** Same procedure via Cloudflare Support.
4. **Domain reset.** Contact the registrar with the same evidence. If the
   registrar is Cloudflare, this happens with step 3.
5. **Resend reset.** Can be abandoned and rebuilt with a new Resend account.
6. **Regenerate all secrets** (§7) and redeploy.

This procedure is handed to the client at delivery in printed form. The
client doesn't need to ask Marc anything to execute it — that's what
"kill-switch" means.

## 9. Acceptance and warranty

- Client signs (paper or DocuSign) an acceptance attestation.
- Marc tags the final commit:
  `git tag v1.0-handoff-accepted-YYYY-MM-DD && git push --tags`.
- **90-day warranty:** Marc fixes any bug introduced by his code, free,
  provided the client tells him within the window.
- Beyond 90 days: Tier 2 hourly rate, only at the client's explicit request.

## Marc's post-handoff cleanup

After the client signs:

- [ ] Remove the client from your 1Password share.
- [ ] Archive the client folder (local + iCloud).
- [ ] Update the engagement's `feature.json` status to `shipped` and write a
      `handed-off` annotation with the client's name + date.
- [ ] If the engagement was in showcase mode, set
      `userGuide` to `"Engagement completed — repo, domain, and accounts
      transferred to <client> on YYYY-MM-DD."`
- [ ] Move the session in `/admin/trash` (soft delete keeps audit history).
- [ ] If this is the first signed engagement: post-mortem of the handoff in
      `bmad/HANDOFF_RETRO_<client>.md` — what worked, what to improve.
